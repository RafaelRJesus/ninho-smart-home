# Objetivo

Garantir que toda funcionalidade seja descrita com comportamento esperado, regras de negócio, exceções e critérios de aceite testáveis.

# Quando usar

Use ao criar ou revisar histórias, épicos, critérios de aceite, regras de negócio e escopo de sprint.

# Entradas esperadas

- Objetivo do produto
- Fluxo do usuário
- Regras de negócio
- Dependências
- Restrições técnicas

# Regras obrigatórias

- Escrever histórias no formato Como/Quero/Para.
- Criar critérios no formato Dado/Quando/Então.
- Incluir cenários positivos, negativos e alternativos.
- Não aceitar requisitos ambíguos.
- Identificar dependências e riscos.

# Saídas esperadas

- História técnica
- Critérios de aceite
- Casos de exceção
- Dependências
- Definition of Ready

# Restrições

Não inventar integrações, permissões ou regras não confirmadas. Marcar toda hipótese explicitamente.

# Checklist de conclusão

- [ ] História clara
- [ ] Critérios testáveis
- [ ] Exceções cobertas
- [ ] Dependências registradas
- [ ] Sem ambiguidade
